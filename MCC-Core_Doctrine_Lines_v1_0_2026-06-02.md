---

## 3. Meaning

### 1. A proposal is not permission.

An autonomous agent, model, workflow, or system may propose an action.

But proposal does not create authority.

The fact that an action was generated, recommended, predicted, planned, or technically possible does not mean it is authorized to execute.

This line separates autonomous reasoning from execution authority.

---

### 2. No verified decision — no execution.

Execution requires a verified decision before action.

MCC-Core evaluates the proposed action and returns an accountable outcome:

- `ALLOW`;
- `DENY`;
- `ESCALATE`;
- `CONSTRAIN`.

Without a verified decision, execution must not occur.

This line defines MCC-Core as a pre-execution decision boundary.

---

### 3. No verified path — no trusted execution.

A decision boundary is valid only if execution-capable systems are structurally required to pass through it.

Agents must not hold direct execution authority.

All execution-capable tools, APIs, infrastructure actions, workflow actions, financial actions, and physical actuators must be reachable only through an `ExecutionGate` that requires a valid `VerifiedDecisionToken`.

If an agent can bypass MCC-Core and execute directly, the system is not MCC-governed.

This line defines bypass resistance and mandatory evaluation path.

---

### 4. No post-factum permission.

Execution must not precede authorization.

MCC-Core rejects post-factum governance as a sufficient control model.

Review, logging, explanation, or remediation after execution cannot replace pre-execution authorization.

If execution happens first and approval happens later, the system is not MCC-governed.

This line defines the Non-Post-Execution Principle.

---

## 4. Strategic Interpretation

These four lines form the public doctrine surface of MCC-Core.

They define the operational shift from autonomous proposal to verified execution governance.

The model may propose.  
The agent may plan.  
The workflow may prepare.  
The system may simulate.

But execution requires:

1. a verified decision;
2. a verified path;
3. a valid execution gate;
4. audit-before-actuation;
5. no post-factum permission.
